Google Authentication¶

Google Authentication protects Shogun Command so only approved Google accounts can access command pages and command APIs. The feature uses Google OAuth for identity, an explicit email allowlist for authorization, and route protection so unauthenticated users are redirected before they reach private dashboard pages.

Overview¶

Introduction¶

Shogun Command is exposed through a public route but is intended for private operation. Google Authentication provides the first security boundary: a visitor must sign in with Google, and the signed-in email must match the configured allowlist before the dashboard or command APIs become available.

This feature keeps the login flow simple for the approved operator while rejecting unknown Google accounts and unauthenticated API requests.

Preconditions

  • Shogun Command is reachable from the browser.

  • Google OAuth credentials are configured.

  • AUTH_ALLOWED_EMAILS contains the approved operator email.

  • The operator is not already authenticated.

Expectations

  • Protected pages are not visible before authentication.

  • Approved Google accounts can enter Shogun Command.

  • Unknown Google accounts are rejected.

  • Unauthenticated command API calls return 401.

Main Flow

  1. The operator opens a protected Shogun Command page.

  2. Shogun Command redirects the browser to /login with the requested page preserved as callbackUrl.

  3. The operator selects Continue With Google.

  4. Google OAuth authenticates the operator.

  5. Shogun Command checks the returned email against AUTH_ALLOWED_EMAILS.

  6. The approved operator receives a session and is redirected to the requested page.

Alternate Flows

  • If the Google account is not allowlisted, Shogun Command returns the operator to /login with an access denied message.

  • If a command API is called without an authenticated session, Shogun Command returns 401 JSON instead of redirecting.

Google OAuth handles identity, then Auth.js returns to Shogun Command so the configured allowlist can decide whether the browser receives a session or an access denied login result. API requests without a session are rejected with 401 JSON by middleware.

Implementation¶

Implementation Overview¶

Google Authentication is implemented as a small set of cooperating Next.js and Auth.js entrypoints. The feature does not put authentication checks inside every page or API handler. Instead, shared Auth.js configuration provides the session helpers, middleware applies the first route guard, and the command layout adds a server-side page guard before rendering private UI.

Code Ownership¶

The implementation centers on features/security/google-authentication/auth.ts. That file creates the Auth.js configuration, registers the Google provider, enables JWT sessions, and exports the helpers used by the rest of the application. The root auth.ts file remains as a thin compatibility export for framework entrypoints and existing imports.

The Auth.js exports are reused in a few focused places:

  • middleware.ts uses auth to inspect the current request session before protected routes continue.

  • app/api/auth/[...nextauth]/route.ts exposes the Auth.js GET and POST handlers needed by the Google OAuth callback flow.

  • features/security/google-authentication/actions.ts uses signIn and signOut for server actions triggered by the login page and command shell.

  • app/(command)/layout.tsx calls auth() before rendering private command pages.

This keeps the authentication rules in one shared configuration while allowing each UI or routing layer to use the helper that fits its job.

Main Implementation Flow¶

features/security/google-authentication/auth.ts reads AUTH_ALLOWED_EMAILS when the application starts and turns the comma-separated value into a normalized set. Each configured email is trimmed, lowercased, and ignored if it is empty. That gives the sign-in callback a simple membership check instead of repeating parsing logic during every login.

The Auth.js configuration registers the Google provider and uses JWT-backed sessions. During sign-in, Auth.js receives the Google profile and the current user object. Shogun Command takes the best available email from those values, normalizes it to lowercase, and only accepts the login when that email exists in the allowlist.

If the email is missing or not allowlisted, the callback rejects the sign-in. Auth.js then sends the browser back to the configured error page, which is the same public /login page used to start authentication.

The middleware is the first application-level gate for incoming requests. It allows public auth paths through, lets authenticated requests continue, and then splits unauthenticated requests by request type.

        flowchart TD
  Request["Incoming request"]
  Public{"Public path?"}
  Session{"Session exists?"}
  Api{"API route?"}
  Allow["Continue request"]
  Redirect["Redirect to /login?callbackUrl=..."]
  Reject["Return 401 JSON"]

  Request --> Public
  Public -- yes --> Allow
  Public -- no --> Session
  Session -- yes --> Allow
  Session -- no --> Api
  Api -- yes --> Reject
  Api -- no --> Redirect

/login and /api/auth/* stay public because they are needed to start and finish the OAuth flow. For every other path, middleware checks whether Auth.js found a session on the request.

When an unauthenticated browser requests a protected page, middleware redirects to /login and stores the originally requested path in callbackUrl. That lets the application return the operator to the page they originally wanted after a successful login.

When an unauthenticated request targets an API route, middleware returns 401 JSON instead of redirecting. The API handlers can therefore stay focused on their domain work; they rely on middleware to reject anonymous callers before the handler runs.

Command pages live under app/(command) and share a protected layout. That layout calls auth() on the server before rendering the sidebar, mobile header, or page content.

Middleware should already have redirected anonymous page requests, but the layout check keeps the page tree defensive. If a page render reaches the layout without a session, it redirects to /login instead of rendering private UI. If a session exists, the layout passes the user into the command shell so the sidebar and mobile header can show the signed-in identity and sign-out control.

app/login/page.tsx is public and intentionally standalone. It does not use the command shell layout, so unauthenticated users only see the focused login screen.

The page reads two query parameters:

  • callbackUrl tells the app where to send an approved operator after login.

  • error lets the page show a friendly access denied message when Auth.js rejects a non-allowlisted Google account.

Before using callbackUrl, the page sanitizes it. Only same-site relative paths are accepted; missing values, absolute URLs, and protocol-relative values fall back to /. If an already-authenticated operator opens /login, the page redirects them to that safe callback path immediately.

features/security/google-authentication/actions.ts contains the server actions used by the login page and command shell. signInWithGoogle receives the submitted callbackUrl, applies the same safe relative URL rule, and starts the Auth.js Google provider flow with that redirect target.

signOutOfGoogle ends the Auth.js session and redirects the browser to /login. The sidebar and mobile header both use this action, so desktop and mobile sign-out behavior stays consistent.

cp .env.example .env.local
npm install
npm run dev

For Pi validation, open https://your-public-hostname.example.com and confirm that / redirects to /login, /login returns 200, Google login loads the dashboard for an allowlisted account, and a non-allowlisted account receives the access denied login error.

Maintenance Notes¶

Keep the allowlist parsing centralized in features/security/google-authentication/auth.ts so every login uses the same normalization rule. Keep /login and /api/auth/* public in middleware.ts; the OAuth flow cannot start or finish if those routes require an existing session.

When adding new command pages or command APIs, keep them behind the existing middleware matcher and command layout instead of adding one-off authentication checks in individual handlers.
